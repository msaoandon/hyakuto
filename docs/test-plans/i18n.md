# Test Plan — Localization (i18n)

Manual test cases for UI localization. Locales: `en` (default), `uk`; UI strings
come from typed dictionaries, read reactively from the store, with `{var}`
interpolation. Selecting a language updates the store (and persists — see
[persistence.md](persistence.md) PERS-02).

Field legend and case-vs-run: see [audio.md](audio.md).

> **Scope note:** only **UI strings** are localized today. Chat **content**
> (messages, choices) is still placeholder/source text and is NOT yet
> translated — see I18N-06.

---

## I18N-01 — Default locale on first launch
- area: i18n
- priority: high
- platforms: [ios, android, web]
- automatable: yes

Preconditions: no saved locale (fresh install / cleared storage).
Steps:
  1. Launch the app.
Expected: UI is in English (`DEFAULT_LOCALE = "en"`); the language chooser shows
ENG selected.

## I18N-02 — Language chooser switches UI strings live
- area: i18n
- priority: high
- platforms: [ios, android, web]
- automatable: partial

Steps:
  1. On the main screen, change the chooser from ENG to UKR.
Expected: visible UI strings (Start, and downstream Choose a day / Reply / Exit)
re-render into Ukrainian immediately, without a reload.

## I18N-03 — Selected locale persists
- area: i18n
- priority: med
- platforms: [ios, android, web]
- automatable: partial

Steps:
  1. Select UKR, quit, relaunch.
Expected: the app reopens in Ukrainian. (Same mechanism as PERS-02.)

## I18N-04 — Interpolation renders variables
- area: i18n
- priority: med
- platforms: [ios, android, web]
- automatable: yes

Steps:
  1. Open the day chooser / a day in each locale.
Expected: `play.day` ("Day {n}") renders with the number substituted (e.g.
"Day 1"), in both locales. No literal `{n}` is shown.

## I18N-05 — Locale key parity
- area: i18n
- priority: high
- platforms: [web]
- automatable: yes

Steps:
  1. Run the i18n parity test (`pnpm test` in apps/web).
Expected: every locale provides exactly the keys defined in `en` — no missing or
extra keys. (Covered by the existing parity test; this case tracks that it stays
green when keys change.)

## I18N-06 — Chat content localization (KNOWN GAP)
- area: i18n
- priority: low
- platforms: [ios, android, web]
- automatable: no

Steps:
  1. Switch locale, then open a chat.
Expected (current): chat message/choice text does NOT change with locale — only
the UI chrome does. This case documents the gap so it isn't mistaken for a bug;
promote to a real expectation once content localization is implemented.
